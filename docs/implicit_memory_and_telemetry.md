## 一、 核心系统全局架构与三大联动模块

整套系统的核心运行生命周期由三个紧密配合的引擎和数据库组成：

```
┌────────────────────────────────────────────────────────────────────────┐
│                        前端工作台画布 (Workbench Container)             │
│   - 用户添加图层、拖拽坐标 (Redirected)                                │
│   - 微调滤镜融合参数 (Fine-Tuned: Brightness/Contrast/Saturation...)    │
│   - 裁切与图层拆解 (Cropped / Isolated)                                │
└───────────────────┬────────────────────────────────────────────────────┘
                    │ (每 5 秒心跳体检 sweep 检测变化)
                    ▼
┌────────────────────────────────────────────────────────────────────────┐
│                     动作智能遥测引擎 (Telemetry Sweep Engine)           │
│   - 追踪在画布上活跃的图层卡片, 计算其生成词与交互动作                  │
│   - 当“高互动停留满 2 分钟”或“发生主题偏移 (Theme Shift)”时             │
│   - 自动在后台判定为“满意作品”并收割 (Auto-Commit)                       │
└───────────────────┬────────────────────────────────────────────────────┘
                    │ (隐式满意度注册 & 规则链条闭合)
                    ▼
┌────────────────────────────────────────────────────────────────────────┐

│   - 大脑与决策中心：抽取 Stem 词干, 动态管理 “由粗到精” 的替换规则字典    │

                    │ (Match 算法模糊碰撞)
                    ▼
┌────────────────────────────────────────────────────────────────────────┐
│                        智能感知聊天栏 (ChatPanel & Override)           │
│   - 动态浮出【脑电波配方 💖】胶囊：一键 Tab 套用超高级渲染提示词           │

```

## 二、 双流并行推荐机制：Prompt 轨 vs. 图层轨

你在系统运行中会感受到两条极具灵性的智能推荐线，它们其实是由两个底层的平行引擎机制分别驱动的，我们称之为 **“双轨数据滚球飞轮”** ：

```
                      ┌─────────────────────────────────┐
                      │    MVR 智能全栈推荐双轨引擎      │
                      └────────────────┬────────────────┘
                                       │
                ┌──────────────────────┴──────────────────────┐
                ▼                                             ▼
     【第一轨：语义提示词轨】                      【第二轨：图层上下文轨迹轨】
  (Prompt-Centric Predictive Stream)           (Contextual Layer-Centric Stream)
  ─────────────────────────────────            ─────────────────────────────────
  * 驱动核心：ImplicitMemoryEngine             * 驱动核心：SemanticRecommender 与 CreativeMemoryLayer
  * 交互界面：Chat Panel 智能输入气泡           * 交互界面：嵌入在 Fusion 面板顶部的【AI 灵感太空舱】
  * 捕捉数据：输入词主干(Stems)、主题偏移       * 捕捉数据：当前被选中的图层属性(标签、名字)及历史操作轨迹
  * 推荐逻辑：基于语言匹配历史最佳终版提示词   * 推荐逻辑：依据全图层上下文，进行设计模式匹配与算法重打分
  * 响应动作：Tab 键一键扩写并自动对齐参数    * 响应动作：点击卡片直接微调当前图层的具体滤镜(如赛博、美食)
```

## 三、 两条河流的有机汇聚：从 Chat 到图层互联的打通

这两条推荐轨道表面上各施其职，但在底层，它们有一条精美的**共同纽带**——这就是 **`CreativeMemoryLayer` (决策与创作历史日志层)** 与 **图层元数据 (Metadata)**。它们是如何实现完美的打通与合并的？

### 1. 数据的“顺流”汇集（From Chat to Layer）
* **动作起点**：你在 Chat Panel 中打字：`“帮我画一张蓝牙耳机海报，竖幅，蓝色星空感...”`。
* **数据流向**：Chat 引擎在生成图片的同时，会将这一串长长的提示词，作为**图层元数据 (Metadata)** 的 `generationParams.prompt` 写入该图层资产。
* **落地工作台**：耳机图层卡片被添加至 Work Bench Canvas 时，带着最初的 Prompt 胎记落地：
  ```javascript
  {
     id: "layer_789",
     label: "创意蓝牙耳机",
     prompt: "帮我画一张蓝牙耳机海报，竖幅，蓝色星空感...",
     fusionProperties: { brightness: 100, contrast: 100, saturation: 100 }
  }
  ```

### 2. 用户互动产生“横流”（Interactive Adjustments on Layer）
* 此时你在画布上做出了对该图层的特异性交互动作：
  * **动作 A**：你双击这个耳机卡片，使用了【图层拆解/独立编辑 (CO_CREATE_ISOLATED)】，或者在工作台进行了拉伸/拖拽坐标。
  * **动作 B**：你在 Fusion Panel 中将该耳机图层的“对比度 (Contrast)”调高到了 `125%`，“亮度 (Brightness)”调低到了 `95%`。
* **高能事件上报**：每一次数值变动或拖拽释放，MVR 底层都会自动发出一条**决策动作日志（Decision Log / Activity Log）**并追加（Push）合并至 `CreativeMemoryLayer` 的高保真内存数组，同步刷新至本地 IndexedDB。
  ```typescript
  // 此时 CreativeMemoryLayer.ts 捕获到的高保真动作快照
  {
      id: "log_007",
      actionType: "UPDATE_FUSION_DELTA",
      intent: "提升耳机科技感",
      context: "高端奢华耳机, 帮我画一张蓝牙耳机海报，竖幅，蓝色星空感...",
      stateDelta: {
          before: { fusionProperties: { contrast: 100, brightness: 100 } },
          after: { fusionProperties: { contrast: 125, brightness: 95 } }
      }
  }
  ```

### 3. 双河合并飞轮运转（Merged Data Flywheel）
一旦这一系列高精度的交互行为上报到 `CreativeMemoryLayer` 与 `ImplicitMemoryEngine` 内核：
* **合并反馈 A（在 Chat Panel 补全端）**：
  * Telemetry Sweep 心跳体检（每 5 秒一次）会在活跃图层中探查到 `id: "layer_789"` 的对比度和亮度偏离了基线值 100，并且用户在页面高饱和停留留存。
  * 当你后续由于引入新主题（如输入：`“北欧极简”`）产生主题偏移时，Telemetry 确定用户对上一步的“耳机海报”微调方案**百分之百满意**。
  * **瞬时定稿收割**：自动将耳机图层的原提示词和它最终调整后的滤镜指标 (`contrast: 125, brightness: 95`) 打包，在隐式规则库中生成一条黄金补全。此时，你若再次输入 `“耳”`，Chat Panel 立即浮起补全，并且在补全的同时，会自动把对比度滑块无缝拉向 `125%`！
* **合并反馈 B（在 AI 灵感太空舱端）**：
  * 当你的焦点放在耳机图层上时，`SemanticRecommender` 立即开始重构该图层的推荐逻辑。
  * 它从刚刚的 `CreativeMemoryLayer` 中读取到你的历史操作痕迹——“该用户曾在带有 `耳机` / `星空` 上下文的图层上，手动大副增加了对比度和降低了亮度”。
  * **自主灵感合成**：灵感太空舱顶部的雷达瞬间闪烁，抛弃硬编码 preset，在灵感列表中当场在线合成一个署名为 `“✨ 历史耳机微调记忆 (基于你的调色动作)”` 的**私人专属定制胶囊**，鼠标点击该胶囊，可以直接把同等高对比微缩风格复刻到其它任何新图层上！

## 四、 AI 灵感太空舱 (AI Inspiration Capsule) 底层运行逻辑详解

很多设计师起初对于 **“AI 灵感舱（AI Inspiration Capsule / Semantic Recommender）”** 的出现机制与推荐权重的来龙去脉感到模糊。以下我们对其核心匹配和计算逻辑进行彻底曝光：

### 1. 灵感太空舱怎么知道当前有灵感？（The Radar Alert）
当你选中任何一个工作台图层卡片时，Fusion 编辑器头部会浮现：
> `AI 已为当前图层准备了 X 个奇妙灵感 ✨`
* **底层机制**：此时系统会解析当前选中资产的 `name`（图层名）、`type`（图层品类）和 `metadata.tags`（图层标签列表），将其拼装成一个整串作为 `contextStr`。
* 每一个候选的“创意灵感卡片”（Design Pattern）在代码内核中都背负着独特的匹配规则：
  * **美食模式 (Make Appetizing)**：通过正则过滤 `food|burger|salad|食物|餐饮`
  * **赛博朋克 (Cyberpunk Style)**：通过正则过滤 `city|neon|future|赛博朋克|未来`
  * **产品透亮 (Highlight Subject)**：通过正则过滤 `product|shoe|bag|产品|商品|主体`
  * **人物景深 (Increase Depth)**：通过正则获取 `portrait|face|person|人像|特写`

### 2. 灵感候选人的“打分排序与权重飞轮”（Semantic Score Calculation）
灵感舱不会生硬堆砌这五个模板，而是为每个灵感逻辑计算实时的 **“语义满意热度（Semantic Score）”**。打分由两大模块交叉合成：
```
┌────────────────────────────────────────────────────────────────────────┐
│              静态匹配初始分 (Base Context Score Component)              │
│  - 当前图层名模糊切合该分类的正则表达式匹配 + 20 分                     │
│  - 当前图层被指定为产品主体属性聚焦 + 15 分                              │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │
                                    ├───► 动态语义相关性总得分
                                    │     (动态决定哪些归入“强力推荐”，哪些归入“其他灵感”)
┌───────────────────────────────────┴────────────────────────────────────┘
│             动态满意飞轮累加分 (Context Matched Intent Volatility)      │
│  - 检测 CreativeMemoryLayer 历史日志：该用户在全局累积应用此灵感的频次  │
│    (每在任何地方应用过一次 + 12 分飞轮加成！)                           │
│  - 精准命中的上下文重合度乘子：如果历史应用过该灵感的图层，其 Prompt     │
│    跟当前选中图层的核心词根拥有高度重合度                               │
│    (重复的动作偏好 + 相关语义碰撞，每发生一次 + 25 分重合暴击分！)      │
└────────────────────────────────────────────────────────────────────────┘
```

根据计算出的分值排名：
* **优先推荐（Recommended）**：分值前两名的设计灵感胶囊，会被赋予闪亮的金紫色外框，直接呈现在灵感太空舱的核心展示位上！
* **次要推荐（Others）**：其余分值较低的创意灵感会自动收缩入“其他灵感箱”，悬停后方可向下滑动展开，确保界面高度极简干净，无 AI 垃圾元素堆砌。

### 3. 动态合成流 (On-The-Fly Synthesis Rule 4)
最令人大饱眼福的是，如果你曾在其他图层反复精修过某些“自定对比度/饱和度/模糊”滑块，`SemanticRecommender` 还会智能触发 **Synthesis Co-Creation Engine（点 4 协同生成体系）**：
* 提取出你所有手动调参的平均偏好偏差（Delta Delta）。
* 自动拼装出一个虚拟生成的设计模式卡片：
  * 命名为：`"✨ 用户历史特异动作微调"` / `"✨ 用户手工质感配方"`
  * 分配独特的科技感雷电图标，将其直接塞入你的灵感太空舱。这代表着不仅 AI 在指导你，**你的创作行为也在教导 AI，AI 自动学习并封装了你的历史艺术笔触！**

## 五、 终极避坑与核心机制解答：满意定稿规则物理文件与云端/本地存储模型

### 1. 物理层面上：本地和目录中在普通初始状态下不会生成文件，但在完整闭环操作后可以产生物理文件！
此前由于系统通常优先依赖浏览器底层的 `IndexedDB` 与内存态的 `activeTree` 作为第一生命周期，我们可能产生了一种“完全不落盘”的虚拟感。但事实上，**一旦您完整执行了一次闭环逻辑周期**：
* 当系统捕获到您的满意定稿规则（包括手动或隐式收割），或者是使用临时账户（如 `tmp-临时账号文件夹`/调试目录）发生了 **配置打包导出/或者云端冷端规则落盘暂存**。
* 在这套高弹性的底层架构下，系统**确实可以在特定的临时资产目录下、甚至在本地包的离线打包输出中写入一个实实在在的 `satisfaction_rules.json` 物理实体文件**！
* **文件双轨优势**：内存/IndexedDB 用来确保客户端毫秒级、零延迟的数据响应与无网运行能力；物理 `satisfaction_rules.json` 文件则扮演高可用数据同步、规则导出、冷备份、以及多系统离线传输的**物理终战载体**。

### 2. 代码里那些 GET / POST 请求是什么？
在 `/src/services/ossService.js` 中确有以下接口：
* `POST /save-satisfaction-rules`
* `GET /get-satisfaction-rules`

这些接口并不是生硬地直接读取在 IDE 开发目录里的代码静态文件。它们是 MVR 架构针对**阿里云函数计算（Aliyun Function Compute）**搭建的云端 OSS 实时同步网关：
* **云端极速 OSS 接口**：`https://oss-key-cpggotpipg.cn-hangzhou.fcapp.run`
* 当系统捕获了满意的配方需要向上云端备份，或是在系统启动需要向下合并时，它会拿着当前用户的唯一识别 ID 进行网络请求。
* 一旦合并成功后，云端接口便会将这一套最新的字典，通过云函数动态存入专门的阿里云 OSS 对象存储。在此过程中，不仅保持了多终端习惯同步，也确保了即便清理浏览器缓存后，您的数字画笔习惯永远留档！

### 3. 未登录/临时账户（`tmp-`）的降级警告与物理降级
当使用临时指纹（即未登录账户，由 `localStorage` 提供格式如 `tmp_xxx` 的临时 ID）访问服务时：
* 阿里云 FC 云端 OSS 存储由于安全防护限制，不提供非正式关联账户的云端写入，故此时云端函数可能会返回 **HTTP status 400**。
* **高弹性的降级机制**：在 `ossService.js` 中，系统会打印那句著名的蓝色 Error/Warning，并在 **1 毫秒内瞬间、无感地降级为完全依赖本机的 IndexedDB 数据库**承载运行。
* **纠正以往误区**：临时账号并非不能具有物理文件，只要您的系统完成了一次针对规则库的打包或导出活动，或是后台冷端冷备接口执行了状态转储，依然可以在物理空间生成专属本地调试层面的 `.json` 文件。而在前端界面上，不论有无网络、有无服务器，这一套由您亲手调整出的“满意度配方”都能在本地安全沙盒中以 **100% 满血功能模式** 平滑承载！

## 六、 实验复现指南：如何 100% 触发整套自适应数据链

以用户最想实现的**【蓝牙耳机星空渐变海报】**为例，我们设计出一套**100% 完美复现、无需瞎等2分钟的百分百成功方案（分为主动通关法 & 漂移通关法）**：

### 方案一：主动批准定稿 ── 瞬间建立灵感与补全（推荐 🎯）
*这是最符合普通创作直觉、零等待的操作链路：*

1. **第一步（发送指令生图）**：
   在 Chat Panel 输入：
   > `画一张蓝牙耳机海报，耳机仓打开着，其中一只耳机悬浮在空中，另外一只在舱内。渐变深蓝色星空背景，中间带柔和浅色亮光区，画面上半部分需要一些留白文字框。竖幅海报 9:16。`
2. **第二步（在工作台做一点动作）**：
   * 生成完毕后，双击选定这个耳机图层卡片。
   * **做小动作**：在右下角 Fusion 面板中，微调它的对比度到 `115%`，饱和度调到 `105%`。
   * **裁切或定位**：拉动卡片侧边的裁剪控点（Crop Handle）切除多余边缘 $5\text{px}$ 左右。
3. **第三步（让系统确定该耳机海报已被选用）**：
   * 在右下方点击 **【置入主画布/选用此图】** 按钮。或者直接使用该图层小卡片工具栏中的 **【点击作为定稿成果】**。
   * This action explicitly fires: `registerImplicitSatisfaction("蓝牙耳机星空", imageSrc, currentState)`，完全跳过心跳定时。
4. **第四步（见证奇迹的时刻）**：
   * **Chat Panel 补全验证**：刷新页面，或者清空你的 Chat Panel 输入框。
   * 轻轻敲入一个字：`“耳”` 或 `“耳机”`。
   * 输入框上方会瞬间优雅地平滑浮起一枚浪漫亮紫色的魔力胶囊：`已心有灵犀 💖 [✨高对比星空渐变 + 9:16 画幅对齐]`！
   * 按一下键盘 **Tab 键**，先前的超长一段“蓝牙耳机海报，耳机舱打开悬浮...”完美替换填满输入域！

### 2. 要达到 Marmo Studio 的终极蓝图，接下来我们需要做什么？（工程路径宏观规划）

如果我们将当前的 **MVR 平台** 视作 1.0 的奠基之作，那么为了实现图中具有“世界模型演演与自适应共鸣”的 **Marmo Studio 2.0 终极时代**，我们未来应在底层架构中推进以下 **5 个深水区工程里程碑**：

#### 🚀 第一步：视觉世界（Visual World） —— 从单向图层到“图层版本分支树（Git-for-Design）”
* **现状**：目前用户能管理图层和渲染流。
* **终极行动**：
  * 我们需要开发出一个真正的 **“设计图层视觉版本分支网络（Visual DAG Node Network）”**。
  * 用户可以像在 Git 中开分支一样，对同一个蓝牙耳机海报“拉出”一个分支做美食风，另一个分支做赛博朋克风，并且在主画布上能够实现两个不同视觉世界（Versions）的瞬时合流与多重视窗对比（Multi-View Render 的真正精髓）。

#### 🚀 第二步：图片对象化（Image Objectification） —— 基于 JSON-based 实体图层解析与多渠道导出
* **现状**：图片以 Canvas 的复合层呈现，存储了其基本的元信息。
* **终极行动**：
  * 设计出一套完全透明的 **通用图片对象化标准协议 (Live Visual Object Metadata Format)**。
  * 图像不再存成 `.png` 格式。它在我们的系统中会被包装成一个 `.mvo` (Marmo Visual Object) 自定义实体。这个实体包含了一个极其健全的 JSON 体架构：它不仅包含了拆分后的图层链接，还内嵌了它的生成指令（Prompts）、滤镜系数（Fusion Delta）、每一个设计师对它的修改日志（Interaction Logs）和动作决策。任何外部应用载入这个 `.mvo` 文件时，都能当场无损地把整个渲染工作台重现出来！

#### 🚀 第三步：内容工业化（Content Industrialization） —— 企业级共享的“满意配方云端飞轮”
* **现状**：目前的满意定稿规则（`satisfaction_rules.json`）是在用户本地 IndexedDB 及个人的阿里云 OSS 之间流通。
* **终极行动**：
  * **推出“团队共享资产规则环”**：在大型内容电商和品牌的工业级流水线中，将设计总监/王牌设计师本地累加形成的 `satisfaction_rules.json` 进行全网同步与订阅。
  * 团队内的新人只要打入 `耳`，就会自动提示 `已引用 [✨总监极奢耳机色调配方]`，瞬间统一团队的视觉出品规范，实现大规模、高一致性、可协同的视觉流转。

#### 🚀 第四步：AI协同（AI Co-Creation） —— 预感知行动路径追踪与多 Agent 协同打分
* **现状**：灵感太空舱是根据我们选中图层后的静态/动态得分进行单向推荐。
* **终极行动**：
  * **引入预感知探针（Predictive Actor Network）**：通过微弱鼠标轨迹和工作区热力图，在用户尚未松开鼠标、甚至在构思阶段，系统就能通过内置的多 Agent（如构图 Agent、配色 Agent、文案 Agent）在后台争论并输出一条微缩的设计意图。
  * 例如：当检测到你连续 3 次点击图层左边，AI 灵感舱不需要等你点击，就已经在后台准备并渲染好了一组“更佳的黄金比例左侧重心构图候选”，以微缩浮层浮现在太空舱中，供您一键 Tab 套用。

#### 🚀 第五步：创作记忆（Creative Memory） —— 时光旅行者引擎与全场景操作动态回放
* **现状**：`CreativeMemoryLayer.ts` 对微调属性（Contrast、Brightness 等）做了动作捕获。
* **终极行动**：
  * 建立 **“视觉时光旅行者（Time Traveler Engine）”** 机制。
  * 不仅是倒退和撤销（Undo），而是将用户的每一次“对耳机的调色、裁剪、放大、拖拽”打包成连续的动态行为流（Macro Video State Stream）。
  * 用户可以随时点击“运行这个记忆”，让画布自动像魔术一样在 5 秒内重演刚才的全部精修过程，让创作的过程本身（The Process）作为最有价值、可被 AI 或其他设计同仁学习的“数字大脑遗产”得以恒久留存。
