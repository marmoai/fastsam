# 应用云端漫游与数据恢复机制说明文档

## 1. 问题背景：为什么清理缓存会导致数据丢失？

在修复之前，应用的存储架构存在一个“断层”。  
 
*   **短期记忆（本地 IndexedDB）**：用户在画板上的操作、聊天记录（Sessions）、保存的素材（Assets），主要存储在浏览器的本地缓存（IndexedDB）中。
*   **长期记忆（云端 OSS）**：虽然应用具备将数据同步到阿里云 OSS 的能力，但在**每次应用开机启动时，它只检查“短期记忆”（本地缓存）**。

**导致数据丢失的根本原因：**
当用户无意中清理了浏览器缓存、使用了无痕模式，或者更换了一台新电脑时，本地的“短期记忆”被清空。旧版代码在启动时发现本地数据库为空，就会直接展示一个空的工作台，**它没有机制去主动从云端（OSS）把“长期记忆”下载回来**。

--- 

## 2. 核心修复：打通云端恢复的“断层”

为了实现“清理缓存也不丢数据”的云端漫游能力，我们对应用的启动流程和数据同步机制进行了核心修复。

### 2.1 修复会话（Session）的恢复断层

在 `src/main.js` 的应用启动流程中，加入了自动检测和恢复逻辑。

**工作原理：**
1.  应用启动时，检查本地 IndexedDB 中的会话列表。
2.  如果发现本地会话列表为空（例如刚清理了缓存）。
3.  自动调用 `dbHelper.restoreSessionsFromCloud()`。
4.  从 OSS 拉取用户之前保存的所有会话元数据和详细状态（包括聊天记录、画板布局等）。
5.  将这些数据重新写入本地数据库，并加载最近的一个会话，恢复用户的工作状态。

### 2.2 补全资产库（Asset Library）的云端同步与恢复

原本资产库是纯本地的，清理缓存就会永久丢失。我们为其补全了完整的云端同步闭环。

**实施步骤：**
1.  **前端同步逻辑 (`src/core/session.js`)**：为 `dbHelper` 添加了 `syncAssetsToCloud` 和 `restoreAssetsFromCloud` 方法。现在每当保存或删除资产时，都会自动防抖同步到云端。
2.  **前端接口调用 (`src/services/ossService.js`)**：新增了 `saveAssetsToOSS` 和 `getAssetsFromOSS` 两个请求函数，用于与后端通信。
3.  **后端接口支持 (`oss.js`)**：在后端服务代码中，新增了 `/save-assets` 和 `/get-assets` 两个路由接口。这两个接口负责将资产库的 JSON 数据存取到 OSS 的 `images/projects/{userId}/assets.json` 路径下。
4.  **启动时恢复 (`src/main.js`)**：在应用启动流程中加入资产库的恢复逻辑。如果本地资产库为空，会自动尝试从云端拉取并重建。

## 3. 常见问题与异常处理

### 3.1 报错：`400 Bad Request` (获取资产列表失败)

**现象：**
控制台报错：`从 OSS 获取资产列表出错: Error: 从 OSS 获取资产列表失败`，网络请求 `/get-assets` 返回 400 状态码。

**原因分析：**
前端代码已经更新，开始向后端请求 `/get-assets` 接口。但是，**部署在阿里云 FC（函数计算）上的真实后端代码尚未更新生效**。
旧版后端不认识 `/get-assets` 这个路由，导致请求一路“漏”到了文件底部的通用文件上传逻辑。上传逻辑因为没有收到文件，抛出了“未收到有效文件”的错误，最终返回 400 Bad Request。

*(注：在阿里云 FC 中，有时仅“部署代码”不够，还需要“发布版本”或“更新别名指向最新版本”才能让外网 URL 真正运行新代码。)*

**解决方案：**
确保将包含 `/get-assets` 和 `/save-assets` 接口的最新 `oss.js` 代码成功部署并发布到阿里云 FC。部署生效后，报错会自动消失。

### 3.2 新用户一开始没有资产，会报错吗？

**现象疑问：**
新用户第一次打开应用，云端 OSS 里确实没有 `assets.json` 文件，前端去请求 `/get-assets` 时会报错吗？

**解答：完全不会报错。**

**原因分析：**
在后端的 `oss.js` 代码中，针对这种情况做了专门的拦截处理：
```javascript
} catch (e) {
  if (e.code === 'NoSuchKey') {
    // 如果是新用户，云端还没文件，不要报错！
    // 直接返回 200 成功状态码，并告诉前端：资产列表是空的 []
    res.writeHead(200, headers);
    res.end(JSON.stringify({ status: 'success', assets: [], message: '未找到保存的资产列表' }));
  } else {
    throw e;
  }
}
```
当 OSS 抛出 `NoSuchKey`（找不到文件）错误时，后端会将其捕获，并向前端返回一个正常的 `200 OK` 响应，包含一个空的数组 `[]`。因此，新用户打开应用时，控制台不会有任何红色报错，应用会平稳启动。

## 4. 总结

经过上述修复，应用现在已经具备了完善的**云端漫游**能力。无论是会话记录还是资产库，只要成功同步到了云端，即使用户清理了浏览器缓存或更换了设备，应用也能在下次启动时自动从云端拉取数据并恢复本地状态，彻底解决了数据丢失的隐患。
