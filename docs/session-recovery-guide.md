# 会话数据丢失与找回指南 (Session Recovery Guide)

## 1. 问题现象
在使用 AI Studio 预览环境开发时，有时重启电脑、刷新浏览器或重新打开项目后，会发现之前的所有历史会话和工作台数据“消失”了，仿佛变成了一个全新的项目。

## 2. 原因分析
这**不是数据丢失**，而是你的“身份凭证”变了。

目前应用在未接入真实登录系统时，使用的是**基于浏览器本地缓存 (localStorage) 的临时 ID**（例如 `tmp_veqjqwtxvomncgxh5z`）。
- AI Studio 的右侧预览窗口是一个嵌套的 `iframe`。
- 出于安全和沙盒机制，当环境重启或 URL 发生微小变化时，浏览器可能会清空或隔离这个 `iframe` 的本地缓存。
- 当应用读取不到旧的临时 ID 时，就会自动生成一个新的临时 ID。
- 因为云端 (OSS) 的数据是绑定在 ID 上的，新 ID 自然看不到旧 ID 的数据。

## 3. 关于真实账号 (终极解决方案)
**如果应用接入了真实的账号系统（如 Authing、Firebase Auth、Google 登录等），就不会出现这个问题。**
因为真实账号的 ID 是固定的（绑定在你的邮箱或手机号上）。无论你换哪台电脑、清空多少次缓存，只要重新登录，应用就能拿到你唯一的真实 ID，并从云端拉取属于你的所有数据。

## 4. 如何找回“丢失”的临时 ID 数据？

如果你还没有接入真实登录，可以通过我们预留的**“后门 URL 参数”**来强制找回数据。


### 操作步骤：
1. **获取旧的临时 ID：**
   - 如果你记得，直接使用。
   - 如果不记得，可以登录你的阿里云 OSS 控制台，查看 `sessions/` 目录下的文件夹名称，那些 `tmp_...` 开头的文件夹名就是你的历史 ID。

2. **获取应用的真实运行地址 (Shared App URL)：**
   - 不要使用带有 AI Studio 侧边栏的嵌套链接（`https://aistudio.google.com/...`）。
   - 使用应用底层的真实独立域名（通常是 `https://ais-pre-...run.app`）。

3. **拼接找回链接：**
   - 在真实地址后面加上 `?force_user_id=你的旧ID`。
   - **示例：** `https://ais-pre-nf4amshghwl4d4i544ff43-57810270001.asia-northeast1.run.app/?force_user_id=tmp_veqjqwtxvomncgxh5z`

4. **在新标签页中打开：**
   - 在浏览器中打开这个拼接好的链接。
   - 页面加载时，底层代码会读取 `force_user_id` 参数，强制覆盖你当前的本地缓存。
   - 稍等片刻，系统就会自动从 OSS 云端拉取该 ID 下的所有历史数据。

5. **后续使用：**
   - 找回成功后，该独立域名的浏览器缓存就已经记住了这个旧 ID。
   - 以后你只需直接访问 `https://ais-pre-...run.app`（无需加参数），数据也会一直存在。
