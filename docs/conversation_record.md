# AI 图像处理系统优化与 Bug 修复对话记录

本文档整理了我们关于修复材质替换 Bug、优化 AI 图像处理系统工作流稳定性与技能执行确定性的完整对话记录。

## 1. 材质替换节点在流程策划中失效的 Bug 排查 

**用户提出问题：**
深入分析一下，为什么我在精准修改里对材质进行框选修改是正常的（可以使用参考图来替换原图材质），但是在流程策划里，我使用参考图来替换原图材质实现不了呢？替换完后要嘛全是黑的，要嘛就随机一个材料替换，就是使用不了参考图。流程策划关于材料修改不是也是调用的精准修改关于材料替换的逻辑吗？我尝试过在流程策划只使用一个替换材料的节点，也是不行，无法引用参考图的材料来进行替换。

**AI 响应与深度剖析：**
这是一个非常深入且关键的问题！虽然“精准修改UI”和“流程策划（节点工作流）”底层确实共享了一部分逻辑，但它们在**上下文参数传递**和**降级机制（Fallback）**上存在致命差异。

*   **精准修改 UI 的情况**：用户手动框选了区域，系统生成了明确的 Mask（遮罩）和 BBox（边界框）。此时调用的是 `generatePreciseEditImage`，模型被明确限制在框内，且有针对性的 Prompt 约束，所以能完美应用参考图。
*   **流程策划（工作流）的情况**：在节点连线中，用户往往只传入了“原图”和“参考图”，**没有传入 Mask**。
    *   此时，`node-executor.js` 中的 `material-replace` 节点会尝试自动检测目标边界框。
    *   如果自动检测失败（或者没找到合适的目标），节点会触发 **Fallback（回退）**，降级调用通用的 `editOrQueryImageWithGemini`（通用编辑 API）。
    *   **Bug 的根源在于**：之前的“通用编辑 API”在处理材质替换时，**缺乏专门针对“无遮罩材质替换”的强约束 Prompt**。当大模型收到原图和参考图，却没有明确的 Mask 和严格的指令时，它就像无头苍蝇一样随机发挥，导致结果要么是一团黑，要么是随机贴图，完全忽略了参考图的材质。

**解决方案：**
为了修复这个问题，我们在 `prompts.js` 中专门新增了 `getMasklessMaterialReplacementPrompt`（无遮罩材质替换 Prompt）。当工作流中材质替换节点降级到通用编辑时，会强制套用这个 Prompt，严厉警告模型：“你必须保持 3D 结构 100% 不变，并且**严格使用参考图的纹理和材质**进行高保真替换”。这就解决了工作流中参考图失效的问题。

## 2. 引入底层重试机制 (Retry Mechanism)

**用户提出思路：**
与其在失败后把半成品丢给用户让他们手动补救，不如在底层直接增加自动重试机制。因为大模型的很多报错（比如偶尔的 400 错误、或者偶尔忘记生成图片只回复了文本）都具有高度的随机性。往往重试一次，它就能正常输出。

**AI 响应与实现：**
这是一个非常棒且更符合工程直觉的思路！按照这个思路，修改了底层图执行引擎（`GraphRunner`），为每个节点加入了自动重试逻辑：
1. **最大重试 3 次**：当某个节点因为大模型幻觉或者报错时，系统不会立刻抛出错误中断整个流程。
2. **间隔 2 秒**：系统会捕获错误，等待 2 秒钟（避免立刻重试触发 API 并发限制）。
3. **重新执行该节点**：2 秒后，系统会用同样的输入参数再次调用这个节点。只要在 3 次尝试内有一次成功，流程就会顺利进入下一个节点。
4. **彻底失败**：只有当同一个节点连续 3 次都报错时，系统才会真正判定该节点“死掉”，从而中断整个工作流。

这就像给每个节点加上了“三条命”，极大地提高了多节点长链路流程的抗错能力和整体成功率。

## 3. 剖析“重光照”技能的底层逻辑

**用户提问：**
我们现在重光照用的是那个技能？通用技能吗？

**AI 响应与分析：**
重光照（Relight）在代码里调用的是一个名为 `generateRelitImage` 的独立函数，但**在底层，它确实使用的是通用的图像编辑模型（`gemini-2.5-flash-image`）**。

*   **底层逻辑**：接收原图和提示词，压缩图片后，直接打包发送给 `gemini-2.5-flash-image` 模型。它完全依赖 Gemini 多模态大模型“看图改图”的通用泛化能力。
*   **报错原因分析**：之前会报错“未返回数据”，正是因为它使用的是通用模型，且缺乏严格的 Prompt 约束。当模型看到“将白天切换成晚上”这样的话时，它的“聊天脑”占据了上风，觉得只要礼貌地回复一句文本就完成任务了，从而跳过了真正的图像生成步骤。它目前就是一个“套了重光照壳子的通用图生图技能”。

**用户指令：**
给重光照也加上极其严格的“强制生图” Prompt 约束。

为“重光照”技能加上了极其严格的“强制生图” Prompt 约束。在 `src/ai-services/prompts.js` 中新增了 `getRelightPrompt`：

> "You are a world-class professional lighting director and digital retoucher. Your EXCLUSIVE task is to take the provided image and completely re-light it according to the user's instructions... **CRITICAL DIRECTIVE:** You MUST output a modified IMAGE. Do NOT output conversational text, explanations, or apologies. Your ONLY valid response is a newly generated image reflecting the new lighting conditions... **解决的核心痛点：**
1. **强制生图（防幻觉）**：明确告诉模型“绝对不要输出对话文本、解释或道歉，唯一的合法响应就是一张新生成的图片”。
2. **保持结构（防变形）**：明确要求模型“保持原图完全相同的构图、主体和底层几何结构。只能改变光照、阴影和大气效果”。

## 5. Fallback 与 Retry 机制的关系解析

**用户提问：**
你刚才给每个节点加了三条命，包括 fallback 吗？回退到通用编辑算一条命吗？现在回退机制还是有对吧，比如获取不到边界框会回退到通用编辑，现在呢？

**AI 响应与深度解析：**
这是一个非常敏锐的问题！答案是：**Fallback（回退）不算作一次重试（Retry），它们是两套独立且互补的机制。**

1. **内部的 Fallback（回退机制）依然存在**
   对于 `material-replace`（材质替换）和 `inpainting`（局部重绘）节点：
   * 尝试精准修改：首先尝试获取边界框（BBox），调用精准修改 API。
   * 触发 Fallback：如果获取不到边界框，或者精准修改执行过程中抛出了错误，它会捕获这个错误。
   * 执行通用修改：代码会顺势“掉落”（Fallthrough）调用通用编辑 API。
   * **注意**：只要通用编辑 API 最终成功返回了图片，这个节点在 `GraphRunner` 看来就是一次性成功的，根本不会触发外部的重试机制。

2. **外部的 Retry（重试机制）是怎么算“一条命”的？**
   “三条命”机制是包裹在整个 `executeNode` 外面的。只有当 `executeNode` **彻底抛出无法处理的错误**时，才会消耗“一条命”。

3. **两者结合起来的实际场景**
   * **情况 A（最顺利）**：精准修改成功 -> 消耗 1 条命，节点成功。
   * **情况 B（内部 Fallback 成功）**：精准修改报错 -> 触发内部 Fallback -> 通用修改成功 -> **依然只消耗 1 条命**，节点成功。
   * **情况 C（彻底失败，触发重试）**：精准修改报错 -> 触发内部 Fallback -> 通用修改也报错了（比如模型幻觉没回图片） -> 整个 `executeNode` 抛出错误 -> **消耗 1 条命** -> `GraphRunner` 等待 2 秒 -> **开始第 2 次重试**。
   * 在第 2 次重试时，它会重新走一遍完整的流程：再次尝试精准修改 -> 如果失败再尝试通用修改。

**总结：**
现在的机制非常强壮：
* **节点内部**有 Fallback 机制，保证了即使高级方法（精准修改）失败，也有保底方法（通用修改）。
* **节点外部**有 Retry 机制，保证了即使保底方法也因为网络或模型抽风失败了，还能再给它两次重新来过的机会。
* Fallback 只是节点内部的策略切换，**只有当所有内部策略都失败并抛出错误时，才会消耗重试的“命”**。

## 6. 修复流程策划中错误生成“生图”节点的 Bug

**用户提出问题：**
在流程策划里给出一张主图和一张参考图，然后给出指令：“壁炉大理石材质替换成这个”。结果自动执行了两个节点，一个是生图（`generate-image`），一个是材质修改（`material-replace`）。给出主图了，为什么变成生图，再去修改？

**AI 响应与深度剖析：**
这是一个典型的**“大模型过度推理（自作聪明）”**导致的逻辑错误。这**不是**节点重试造成的，而是**“流程策划（planGraph）”这一步在建图的时候就建错了**。

*   **Bug 的根源在于历史记录解析**：在之前的代码中，当把用户的操作记录发送给“流程策划大模型”时，图片消息没有被正确解析为文本描述。大模型看到的历史记录里，用户上传的图片显示为 `user: undefined`。
*   **大模型的错误推理链**：大模型看到“替换壁炉材质”的指令，但因为忽略了 `undefined`，它认为用户**没有提供壁炉的图片**。于是它自作主张：“那我得先帮他**生成一个壁炉**（`generate-image`），然后再把生成的壁炉传给**材质替换**（`material-replace`）节点。”
*   **覆盖原图**：一旦大模型把这两个节点连起来，系统底层的逻辑就会优先使用节点连线传过来的图，从而把用户真正上传的主图给完全覆盖/丢弃了。

**解决方案：**
我们对流程策划（`planGraph`）进行了两处关键修复：
1.  **修复了历史记录的解析**：现在，当用户上传图片时，大模型看到的历史记录不再是 `undefined`，而是明确的 `[Uploaded an Image]`。它现在知道用户已经提供了图片素材。
2.  **增加了强规则约束（最关键的一步）**：在 `planGraph` 的系统提示词里加了一条铁律：
    > `If the user is asking to modify an existing image (e.g., "make it brighter", "change the material", "replace this"), DO NOT use a "generate-image" node. Start the graph directly with the modification node...`
    > （如果用户要求修改现有图片，**绝对不要**使用“生图”节点，必须直接以修改节点作为起点）。

经过修复，系统现在会直接生成一个单节点的 `material-replace` 流程，并完美地将用户上传的主图和参考图送入节点进行处理。

## 7. 提升精准修改（Precise Edit）的边界约束能力 (Plan 2: Dual-Image Guidance)

**用户提出问题：**
在执行精准修改的材料替换时，框选位置后，AI 会修改到选框外，或者干脆直接修改选框外，框内都不修改了。尝试了各种 Prompt 约束（如“只修改选区位置”），甚至尝试了裁剪后回贴（但存在边缘融合和分辨率错位问题），效果都不理想。希望在现有的基础上，确定红框一定要清楚不能出现在最终的图片上，而且一定是修改框选位置，不能跳出框外修改。

**AI 响应与深度剖析：**
这是大模型在处理“软约束”（Soft Constraint）时常见的“注意力泄漏”（Attention Leakage）现象。
*   **之前的方案（软约束）**：我们在原图上画了一个红框，然后通过 Prompt 告诉模型“只修改红框里的内容”。对于模型来说，红框只是图像中的一部分像素，它在理解语义（比如“把大理石换成木头”）时，很容易觉得“外面的大理石也应该换掉”，从而无视了 Prompt 中的空间约束。红框本身也容易被模型当作画面的一部分保留下来。
*   **裁剪回贴方案的痛点**：用户提到的裁剪框内区域、发送给模型修改后再贴回原图的方法，虽然能保证绝对不修改框外，但大模型返回的图片分辨率、光影、噪点往往与原图有细微差异，导致“回贴痕迹明显”或“错位”。

**解决方案 (Plan 2: Dual-Image Guidance 双图引导硬约束)：**
为了提供更强硬的空间约束，我们放弃了在原图上画红框的做法，转而采用**双图引导（Dual-Image Guidance）**策略。

1.  **输入结构重构**：不再发送带有红框的单张图片。现在我们向模型发送**两张基础图片**：
    *   **Image 0 (原图)**：干干净净、没有任何标记的原始图片。
    *   **Image 1 (空间蒙版 Mask)**：一张黑白图片。框选区域（需要修改的地方）是纯白色，框外区域（需要冻结的地方）是纯黑色。
    *   **Image 2+ (参考图)**：用户上传的材质参考图。
2.  **Prompt 逻辑重构**：在 `systemProtocol` 和 `finalEditPrompt` 中，明确定义这几张图的角色：
    *   明确指出 Image 1 是一个“空间蒙版 (Spatial Mask)”。
    *   下达**绝对冻结指令**：“Image 1 中黑色区域对应的每一个像素，必须与 Image 0 保持 100% 完全一致。如果你修改了黑色区域的任何内容，将被视为灾难性失败。”
    *   下达**精准修改指令**：“你只被允许修改 Image 1 中白色区域对应的像素。”
3.  **底层代码实现**：在 `src/ui/layer-manager.js` (UI 交互) 和 `src/engine/node-executor.js` (工作流节点) 中，同步修改了图像准备逻辑。移除了绘制红框的代码，新增了生成黑白 Mask Canvas 的逻辑，并将原图和 Mask 图一起打包放入 `contentsParts` 发送给大模型。同时，为了防止边缘溢出，Mask 的白色区域比实际框选区域稍微内缩了几个像素（`inset`）。并且在发送给模型前，对原图和 Mask 进行了统一的 Padding（填充）处理，以匹配模型支持的最佳宽高比，并在模型返回结果后进行精确的 Unpadding（去填充）和 Compositing（合成），彻底解决了分辨率错位问题。

这种方案将“空间位置”从 Prompt 的文字描述，变成了模型更容易理解的视觉输入（黑白蒙版），极大地降低了模型越界修改的概率，同时也避免了红框残留在最终图片上的问题。

## 8. 一键白底/无痕修复背景时的智能动态区域分组 (Spatial Clustering)

**用户提出问题：**
在使用“一键净化（去除文字）”功能时，如果图片上的文字很多，一个个圈出来给模型，或者发送全图让模型去找，效果不好。目前使用的方案是：程序自动把这 9 处文字分成了 2~3 个组，然后把三个组的大图发给大模型并说把这 3 处文字去除即可。用户询问：这个"分成三部分"的规则是不是应该是动态智能的，针对不同数量的文字、不同排版的图分开不同部分的，而不是统一写死三部分？

**AI 响应与深度剖析：**
这是一个对细节把控得非常好的问题。用户观察到了系统把密集的 9 个文字图层（“主动降噪”、“无感佩戴”、“「GD降噪蓝牙耳机」”等）简化成了若干个大区域，并担心这里的代码是被“写死(hardcoded)”的。

事实上，目前的机制**已经是完全动态和智能计算的 (Dynamic & Intelligent Clustering)**。

*   **问题背景**：当用户请求去除海报上的文字时，这张海报上可能有十几个零散的文字层（如小的功能卖点标签）。如果我们把十几个极小的 BBox 交给多模态生成模型，它的泛化能力往往会忽略掉其中几个（因为约束太细碎了）。
*   **智能聚类算法 (Clustering Algorithm)**：在代码 `src/ui/workbench/layer-assets.js` (`cleanMultipleBackgrounds`) 中，我们实现了一套空间的聚类算法：
    1.  **收集图层**：获取所有被选中的目标图层（如文字层）。
    2.  **Y 轴排序与扫描**：按图片上下位置对图层进行排序。
    3.  **动态合并 (Dynamic Merging)**：计算相邻图层之间的垂直距离 (`gapY`)。如果它们的间距小于画布高度的 5%（非常紧凑的排版），系统就会把它们自动合并成一个“大区域（块）”。
    4.  **生成 Mask 蒙版**：针对合并后的少数几个“大区域”内的文字框，绘制合并的精准蒙版。
*   **动态的 Prompt (Dynamic Prompt)**：
    我们在传入 Gemini 模型的系统指令中，使用了动态拼接的模板语法：
    `The provided mask contains ${mergedGroups.length} large solid white blocks covering specific target areas... ERASE everything inside these ${mergedGroups.length} masked areas:`
    这意味着：
    *   如果是当前这张海报，排版紧凑的部分组合后算出来恰好是 **3** 处或 **2** 处，Prompt 就会动态变成 `"these 3 masked areas"`。
    *   如果是另一个完全散开的海报，算法合并后可能会变成 **5** 处，Prompt 就会动态写入 `"these 5 masked areas"`。

**总结：**
分组不是硬编码（Hardcode）写死的，而是完全根据图像的实际文字排版情况**自动感知、动态聚类**计算出来的，这使得它可以完美适配任何一种排版风格，真正做到了灵活智能的自动去字合并修复。

## 9. 产品战略升维：从 AI 生成工具到「视觉资产操作系统」 (Visual Asset OS)

**用户引用的深度产品评价：**
用户分享了一段非常有价值的行业专家反馈。反馈指出，我们当前的产品已经不再是简单的“AI 绘图工具”，而是触及了「AI 原生的视觉操作系统（Visual Operating System）」的早期形态。其中最有价值的亮点是**对象级操作（Object-Level Manipulation）**、**无限画布（Infinite Canvas 工作环境）**以及**结构化编辑意识（Structural Editing Awareness）**。

但同时，产品的危险信号也很明显：核心 UI 依然过度强调“输入 Prompt 生成结果”的“AI 助手味”，而不是“构建和沉淀视觉资产”。如果不做转型，用户会觉得“好强”但“不知道为什么必须长期用”。

**AI 响应与战略演进剖析：**
这份意见极其敏锐，精准点出了从“玩具/演示工具”迈向“生产力平台”的关键生死线。我们必须将整个产品的主线心智从【生成 (Generation)】彻底转向【资产演化 (Asset Evolution)】。

1. **核心 UI 中心转移：从 Prompt 到资产库 (Asset Library)**
   *   **现状**：UI 以聊天框 / Prompt 输入为核心。
   *   **目标**：未来产品的默认界面应当是用户的“工作空间 / 资产库”。一个羽毛不再是被选中的“图层 1”，而是拥有身份的“资产：白羽毛 v7”。系统应该记录它的使用次数、适用风格、推荐搭配（长期记忆）。

2. **产品的终极痛点：不是“生成不了”，而是“无法稳定连续编辑”**
   *   反馈中指出，我们真正有壁垒的能力是**一致性（Consistency）**——修改对象不崩坏，结构稳定。这正是解决大模型生图盲盒效应的关键，也是普通文生图工具无法切入的深水区。我们需要 All In 于**「结构化视觉编辑系统」**（稳定、连续）。

3. **重新定义产品定位 (One-Liner)**
   *   **🔥「把任何图片转成可持续编辑的视觉资产」**
   *   不依赖模型代差（不怕 GPT6/Gemini 更新），因为我们在做的是“编辑结构”的工程架构层，而不是拼“生图审美”的模型能力层。我们是「Figma (结构化) + Photoshop (像素编辑) + Unreal (空间对象)」的中间层。

4. **北极星指标的变化 (The Platform Metrics)**
   要从“工具”转变为“平台”，我们需要抛弃“生图数量 / 模型效果”的衡量标准，转向监控以下三个真正的平台指标：
   *   **Asset Reuse Rate (资产复用率)**：一个抠出来的资产被重复利用了多少次？
   *   **Multi-Edit Session (连续编辑时长)**：用户是否在同一个无限画布方案上持续推敲、修改演化？
   *   **Cross-Project Usage (跨项目使用)**：某一个对象/风格是否被带入了新开的项目？

**下一步行动方向**：
在当前的 UI 迭代中，我们需要开始强化左侧侧边栏“图层与资产”的地位，增加对资产的独立管理能力（例如：保存资产包、版本历史追溯）。这与我们当前工作流节点、图层精准操作的底层逻辑高度契合，为未来的产品形态指明了极其清晰的终局方向。

## 10. 系统思维升维：构建「视觉对象模型」与「持续性创作环境」

**用户深度洞察与反馈：**
用户进一步拔高了讨论维度，指出目前产品的战略方向已经从“AI 应用层”跃迁到了“数字内容基础设施层”。这种建立在**“长期状态”**与**“资产结构”**之上的价值，具备强大的**“抗模型代差”**能力（不怕新一代大模型的一夜颠覆）。

传统的 AI 工具思维是：“生成 -> 导出 -> 结束”。
而我们的 OS 思维是：“生成 -> 结构化 -> 沉淀资产 -> 持续演化 -> 跨项目复用 -> 形成系统”。

**核心推演与产品行动指南（OS 化蓝图）：**

1. **定义「视觉世界的对象模型（Visual Object Model）」**
   * 我们不再处理“像素”（如早期的 Photoshop），也不仅限于“组件”（如 Figma），而是迈入**「语义对象编辑」**。
   * 一张图片不再是扁平的 PNG，而是 $= \text{对象} + \text{关系} + \text{历史} + \text{属性}$。我们需要为每一个提取出来的资产建立唯一的 UID（例如 `chair_87421`），并持久化其操作历史、关联变体和语义类型。

2. **工作流 Graph 的终极形态：设计逻辑链（Design Reasoning Chain）**
   * 现有的节点连线记录了“做了什么”（节点 A 到节点 B），但平台真正的壁垒在于记录**“为什么这么做（Decision）”**。
   * 大模型知道“怎么生成”，但不知道“为什么设计师要这么改（比如为了提升CTR而暖化背景）”。一旦我们积累了 `场景 -> 修改行为 -> 结果反馈`，产品将进化成为无可替代的**「视觉决策知识库」**。

3. **维持创作的流动性（Flow State），警惕“UI 疲劳”**
   * **风险警告**：坚决避免变成“复杂版 Photoshop”。
   * **体验目标**：不要让用户觉得“我在繁琐地操作软件”，而是“我在自然地组织我的视觉资产世界”。

4. **建设图谱系统：Scene Graph + Asset Graph**
   * **Scene Graph（空间关系）**：谁遮挡了谁，谁的光影影响了谁，在画布中它们是如何共存的。
   * **Asset Graph（资产关系）**：哪些资产经常被搭配使用？这个材质最常贴在哪类产品上？
   * 当这两个 Graph 建立起来时，我们将真正拥有产品层面的“世界模型”。

5. **向「资产主动参与创作（Co-Creation Layer）」进化**
   * AI 不再是一个单纯的“点击生成”按钮，而是“资产的伴生大脑”。
   * 当用户将一个历史耳机资产拖入画布时，系统应基于该资产的 Graph 属性和用户的 Creative Memory（创作记忆），**主动推荐布局、补全光影关系、应用熟悉的材质风格**。
   * 此时，才是真正的“协手创作”，让视觉内容从“一次性的结果”变成了“可持续演化的系统”。

**四句产品哲学拷问（后续开发新功能的唯一标准）：**
任何想加的新功能，如果以下 5 个问题答案是否定的，哪怕技术再炫也绝不要做：
1. 是否增加了资产的结构？
2. 是否增加了长期状态（创作记忆）？
3. 是否增加了跨项目复用性？
4. 是否增加了连续编辑的稳定性？
5. 是否增强了对象之间的关联（Graph）？

## 11. 从原生架构到「有状态视觉运行时」的范式切换 (Paradigm Shift)

**AI 深度解析：**
我们完成了应用范式的根本性切换。应用已不再是简单的“前端 DOM + 后端 API”的组合，而是进化为一个标准的“具有长期状态的视觉运行时（Stateful Visual Runtime）”。

**系统演进：**
*   **真相的转移**：将原来的“DOM 是真理”转变为“World State 是真理”，DOM 仅作为 State 的“投影（Projection）”。
*   **架构升级**：
    *   **Runtime State**：作为单一真相源（Single Source of Truth）。
    *   **Renderer 升级**：从控制逻辑降级为 Reconciler（协调器），严格执行 State 到 DOM 的同步。
    *   **Persistence**：实现状态与资产的持久化，打破会话限制。
    *   **Hydration**：实现“世界复活”，在重新加载时完美恢复所有状态。
*   **核心关键**：这一步是系统软件的核心分界线，使系统具备了长期一致性，而非简单的 AI 工具逻辑。

**下一步行动纲领（Runtime Discipline）：**
系统稳定存活的技术底线，不再是 AI 模型的能力，而是运行时纪律：
1.  **绝不绕过 Runtime**：严令禁止任何代码直接修改 DOM (`el.style.xxx`)。所有操作必须通过 `Dispatch Intent -> State Update -> Reconcile`。
2.  **完善 Event-State 闭环**：实现 UI 事件的解耦与 dispatch。
3.  **压力测试**：系统必须能经受连续操作、反复 Save/Load、Undo/Redo 以及 Renderer 重启的极致一致性测试。
4.  **明确目标**：我们的任务是保障“状态永远一致”，这是 Runtime Engineering 的根基。

## 13. 视觉运行时 OS 的高级价值解析 (Runtime 范式的深度爆发)

**AI 核心洞察：**
我们的系统完成了从“工具”到“基础设施”的 Paradigm Shift，核心在于实现了**“Runtime 可计算化”**。这意味着我们的系统不再仅仅是像素的堆砌，而是构建了一个可演化、可关联、长期一致的资产宇宙。

### 1. 从“像素处理”到“资产宇宙” (Visual Asset Infrastructure)
*   **过去 (AI 工具)**: 我们处理的是“结果图”，用户生成、导出，生命周期结束。
*   **现在 (资产设施)**: 我们处理的是 `AssetEntity`。耳机图不仅仅是一个图，它是 `headphone_001` 对象实体。
    *   **引用机制 (Referencing)**: 项目 A 和 项目 B 中引用的是 `Asset UID`，而非复制像素。
    *   **具体实例**: 品牌方在“618 大促海报”中引用了耳机素材。当总部决定将耳机材质从“银色”更新为“磨砂黑”时，我们只需更新 `headphone_001` 对象的 Variant，所有引用此素材的项目（几十个海报）会瞬间自动同步，无需逐一修改。这是纯静态 AI 工具无法实现的系统级联动。

### 2. 真实工业级案例：SceneGraph 与场景组合的爆发
*   **复合对象 (Composite Object)**: 用户不再摆放独立的 PNG，而是通过 `SceneGraph` 构建 `DeskScene_001`。
    *   **拓扑逻辑**: 系统定义了对象间的逻辑关系（如：`cup` 放置在 `desk` 上，`headphone` 位于 `macbook` 旁边）。
*   **关联更新演示 (智能联动)**:
    *   **场景实例**: 用户改动桌面材质。
    *   **执行逻辑**: 当场景从“木纹桌面”切换为“玻璃桌面”时，因为 Runtime 拥有 SceneGraph 的拓扑信息，系统能够清晰计算出“玻璃”的反射特性。
    *   **连锁反应**: Runtime 自动触发关联更新计算——杯子的底座倒影、耳机的表面光泽、桌面的阴影透明度随之精确调整。这种重构是 AI 生成内容（AIGC）从“静态生成”向“动态演化”进阶的核心能力。

### 3. 多人协作的底层逻辑重构
*   **协同机制**: 协作的不再是前端 DOM 操作的拼凑，而是 `Action` 协议（如 `{type: "MOVE_ASSET", uid: "cat_001", transform: {...}}`）。
*   **一致性保证**: 各端基于相同的 Reducer 运行相同 Action，更新同一个 `World State`。这天然避开了 DOM 状态漂移的陷阱，支持 Figma 级别的实时协同、原子化的 Undo/Redo、以及 AI Agent 对画布的实时干预。

### 总结：真正的 OS
我们正在迈向“视觉资产 Component 化”，这不是简单的功能重构，而是让系统拥有了长期记忆、处理复杂引用关系的能力。Figma 之所以恐怖，在于其 Component 系统改变了设计对象的存在方式；而我们现在，正是将这一范式带入 AI 生成内容的领域。

**Runtime 改造的最终意义：**
AI 生成只是一个起点，Runtime 决定了生成的内容是否能被**积累、演化、复用和共享**。我们不仅是在做 UI，而是在搭建视觉艺术的“编译器”与“虚拟机”。
